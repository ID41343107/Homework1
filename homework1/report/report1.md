# Problem 1: Ackermann

## 解題說明

## 一. 遞迴

問題描述：Ackermann 函數定義如下 :

| A(m,n)                 | 條件                  |
|------------------------|----------------------|
| n + 1                  | 當 m = 0             |
| A(m-1, 1)              | 當 n = 0             |
| A(m-1, A(m, n-1))      | 其他                 |

求出Ackermann 函數值。

解題策略：

1. 觀察公式：

Ackermann 函數是依賴自身的遞迴呼叫，可直接用遞迴函式，不用額外的資料結構。

2. 設計遞迴函式 A(m, n)：

         若 m = 0，直接回傳 n + 1。

         若 n = 0，呼叫 A(m - 1, 1)。

         否則呼叫 A(m - 1, A(m, n - 1))。

4. 輸入與輸出設計：

       由使用者輸入整數 m 與 n。

       輸出結果為 Ackermann(m, n)。

4. 注意遞迴深度：

   因 Ackermann 函數成長極快，遞迴層數也非常深，因此僅適合小範圍測試（例如 m ≤ 3）。

## 二. 非遞迴

1. 分析遞迴結構

   Ackermann 函數的三種情況分別是：

         若 m == 0，回傳 n + 1。

         若 n == 0，回傳 A(m - 1, 1)。

         否則回傳 A(m - 1, A(m, n - 1))。
   
   這三種情況都可以視為不同的「呼叫」，因此可用堆疊呼叫。

2. 以堆疊模擬遞迴呼叫

         遞迴在程式執行時，本質上就是不斷「將目前的狀態推入呼叫堆疊」，等到子呼叫結束後再回到上一層。

         非遞迴版本可利用陣列手動控制這個過程。

3. 設定初始條件

         將 m 推入堆疊，表示第一次呼叫 A(m, n)。

         n 保留在變數中，用來追蹤目前的參數值。

4. 重複執行直到堆疊為空

   每次從堆疊取出頂端元素 m，根據情況判斷：

   情況一： m = 0

   根據定義 A(0, n) = n + 1，直接令 n = n + 1。

   情況二： n = 0

   根據定義 A(m, 0) = A(m - 1, 1)，將 (m - 1) 推回堆疊，並設 n = 1。

   情況三：其他情況

   根據定義 A(m, n) = A(m - 1, A(m, n - 1))，代表要先處理內層的 A(m, n - 1)。

   因此：

      1. 先將 (m - 1) 推入堆疊（外遞迴）

      2. 再將 m 推入堆疊（內遞迴）

      3. 最後令 n = n - 1，繼續下一輪。

5. 當堆疊清空時結束

   當所有模擬呼叫都完成後，變數 n 即為最終結果 A(m, n)。

## 程式實作

以下為主要程式碼：

## 一. 遞迴

```cpp
#include <iostream>
using namespace std;
int A(int m, int n) {
    if (m == 0) return n + 1; //if m=0,n+1
    else if (n == 0) return A(m - 1, 1); //if n=0,A(m-1,n)
    else return A(m - 1, A(m, n - 1)); //otherwise , A(m-1,A(m,n-1))
}
int main() {
    int m, n;
    cout << "輸入 m 和 n : ";
    while (cin >> m >> n) {
        cout << "A(" << m << ", " << n << ") = " << A(m, n) << endl;
        cout << "輸入 m 和 n : ";
    }
    return 0;
}
```

## 二. 非遞迴

```cpp
#include <iostream>
using namespace std;

int A_no(int m, int n) {
    int s[10000];
    int top = -1;
    s[++top] = m;

    while (top >= 0) {
        m = s[top--];

        if (m == 0) {
            n = n + 1; // 對應 A(0, n) = n + 1
        }
        else if (n == 0) {
            n = 1; // 對應 A(m, 0) = A(m-1, 1)
            s[++top] = m - 1;
        }
        else {   // 對應 A(m, n) = A(m-1, A(m, n-1))
         // 先處理 A(m, n-1)，之後再處理 A(m-1, result)
            s[++top] = m - 1;
            s[++top] = m;
            n = n - 1;
        }
    }
    return n;
}

int main() {
    int m, n;
    cout << "輸入 m 和 n : ";
    while (cin >> m >> n) {
        cout << "A_no(" << m << ", " << n << ") = "<< A_no(m, n) << endl;
        cout << "輸入 m 和 n : ";
    }
    return 0;
}

```
## 效能分析

## 一. 遞迴

## 1. 時間複雜度分析 (Time Complexity)

Ackermann 函數定義為：

| A(m,n)                 | 條件                  |
|------------------------|----------------------|
| n + 1                  | 當 m = 0             |
| A(m-1, 1)              | 當 n = 0             |
| A(m-1, A(m, n-1))      | 其他                 |

觀察公式可以發現：

1.1 遞迴結構高度依賴自身：

對於 m > 0 且 n > 0，每次呼叫 A(m, n-1) 會再呼叫 A(m-1, …)。

隨著 m 增加，遞迴樹呈 指數級甚至超指數級增長。

1.2 時間複雜度特性：

    對於小的 m（如 0, 1, 2）：

        m = 0 → O(1)

        m = 1 → O(n)

        m = 2 → O(2n)

        m = 3 → O(2^(n+3) - 3)

    一般情況下，時間複雜度 無法用多項式表示，會隨 m 與 n 快速爆炸，通常稱為 Ackermann 增長階。

總結：

| m 值	| 大概時間複雜度 |
|-------|---------------|
| 0  	| O(1)          |
| 1    	| O(n)          |
| 2	    | O(n²)         |
| 3	    | O(2^n)        |
| ≥4    | 超指數增長 (非常大) |

因此對程式碼而言，時間複雜度非常高，即便 n 很小，m 大於 3 就會造成大量遞迴呼叫。

## 2. 空間複雜度分析 (Space Complexity)

空間複雜度主要由 遞迴堆疊 (call stack) 決定：  

   2.1 遞迴深度：

    最深的遞迴層數大約等於 A(m, n) 本身的值。

    由於 Ackermann 函數增長極快，遞迴深度也會爆炸。

   2.2 堆疊使用：

    每個遞迴呼叫都需要存放函式參數與返回地址。

    對小的 m（0~3）可接受，但對大 m（≥4）會導致 Stack Overflow。

總結：

| m 值	| 大概空間複雜度 |
|-------|---------------|
| 0  	| O(1)          |
| 1    	| O(n)          |
| 2	    | O(n²)         |
| 3	    | O(2^n)        |
| ≥4    | 超指數增長 (容易 Stack Overflow) |

## 二. 非遞迴

## 1. 時間複雜度分析 (Time Complexity)

Ackermann 函數定義如下：

| A(m,n)                 | 條件                 |
|------------------------|----------------------|
| n + 1                  | 當 m = 0             |
| A(m-1, 1)              | 當 n = 0             |
| A(m-1, A(m, n-1))      | 其他                 |

   1.1 非遞迴執行特性

   非遞迴的核心是：
      
      透過手動模擬遞迴呼叫，利用陣列或變數暫存當前狀態，逐步還原遞迴運作。

      此方式不會減少計算次數，只是將系統堆疊改成程式內部控制流程，因此時間複雜度與遞迴版本相同。

   1.2 時間複雜度近似分析
   
| m 值 |	大概時間複雜度 | 說明                |
|------|---------------|--------------------|
| 0    |	O(1)          |	直接回傳 n + 1    |
| 1    |	O(n)          |	相當於線性增長     |
| 2    |	O(n²)         |	雙層迴圈結構       |
| 3    |	O(2ⁿ)         |	指數成長          |
| ≥4   | 超指數級       |	幾乎無法實際計算   |

 結論：
      
    非遞迴寫法的時間複雜度雖與遞迴相同，但能避免函式呼叫開銷。
      
    執行速度略快，但仍不適用於大 m 值。

## 2. 空間複雜度分析 (Space Complexity)

   2.1 非遞迴的記憶體使用

      非遞迴不使用系統呼叫堆疊，而是以程式內部變數或手動堆疊模擬。
      
      若用固定大小的變數紀錄階層深度，則空間複雜度可視為 O(m) 或 O(m + n)。

   2.2 比較和遞迴不同
   
| 實作方式 |	主要佔用來源 | 空間複雜度| 特性 |
|---------|-------------|----------|-----|
| 遞迴版本 | 系統呼叫堆疊 | O(2ⁿ)（極深）  | 可能造成 Stack Overflow |
| 非遞迴版本 | 手動控制變數/陣列 | O(m+n) | 安全性高，不易爆堆疊 |

結論：

   非遞迴法能有效避免遞迴堆疊過深的問題，雖然整體計算量相同，但執行更穩定。

## 三. 效能結論

   效能結論比較

| 分析面向 | 	遞迴 | 	非遞迴 |
|---------|------------|-------------|
| 實作方式 | 系統遞迴呼叫 | 	以變數或陣列模擬 |
| 時間複雜度 | 	超指數級 | 	超指數級 |
| 空間複雜度 | 	極高（依遞迴深度） | 	較低（O(m+n)） |
| 穩定性 | 	容易 Stack Overflow |	穩定，不會爆堆疊 |
| 效率 | 函式會帶來額外系統負擔 | 稍快一點 |

## 申論及開發報告

## 一. 遞迴

使用遞迴來做Ackermann 函數的原因如下：

1. **程式邏輯直觀**  
   遞迴能清楚表達 Ackermann 函數 的核心概念： 
   當 𝑚=0時，直接返回 𝑛+1 。

   當 𝑛=0時，呼叫 𝐴(𝑚−1,1) 。

   否則，呼叫 𝐴(𝑚−1,𝐴(𝑚,𝑛−1)) 。

   這種方式與函數的數學定義一致，程式邏輯直觀，容易理解。
2. **易於理解與實現**  
  遞迴程式碼與數學公式接近，能直接反映 子問題分解 的思想。
  範例程式碼：
   ```cpp
   int Ackermann(int m, int n) {
    if (m == 0) return n + 1; //if m=0,n+1
    else if (n == 0) return Ackermann(m - 1, 1); //if n=0,A(m-1,n)
    else return Ackermann(m - 1, Ackermann(m, n - 1)); //otherwise , A(m-1,A(m,n-1))
   }
   ```
   這樣的寫法簡潔，無需額外變數來維護中間狀態。
  
3. **遞迴清楚**  
   每次遞迴呼叫對應一個「子問題的解」，最終返回結果會逐層計算完成整體問題。
   
   這種設計讓程式結構簡單，邏輯明確，易於維護。

4. 注意事項
   遞迴深度過大可能導致 Stack Overflow。

   對於大數值的 m 或 𝑛，可考慮改用迭代或其他記憶化方法以節省堆疊空間。

總結：

透過遞迴實作 Ackermann 函數，不僅程式簡單明瞭，也能清楚呈現「子問題分解」與遞迴核心思想，是展示遞迴概念的理想範例。

## 二. 非遞迴

使用非遞迴方式實作 Ackermann 函數的原因如下：

1. 避免遞迴堆疊溢位 (Stack Overflow)

         Ackermann 函數的遞迴層數會隨著 m 與 n 急遽增加，系統堆疊空間有限，若遞迴層數過深會導致 Stack Overflow。
   
         改用非遞迴方式能以迴圈與變數模擬遞迴過程，避免堆疊空間不足的問題。

2. 消除函式呼叫開銷

         遞迴版本在每次呼叫函式時，都需建立新的執行環境，儲存參數、回傳位址等資訊，這些額外動作稱為函式呼叫開銷。

         非遞迴版本透過迴圈控制流程，直接在同一層執行邏輯，能省去函式呼叫與返回的時間，執行效率略高。

3. 以迴圈模擬遞迴邏輯

   非遞迴實作的核心思想，是以變數模擬遞迴的「呼叫與回傳」過程：

         當 m = 0 時，回傳 n + 1。

         當 n = 0 時，m 減 1，n 設為 1。

         否則，將 (m-1) 暫存，繼續計算 A(m, n-1)。
   
   透過迴圈重複執行上述邏輯，即可逐步還原遞迴過程。

4. 程式結構簡單，穩定性高
   
   非遞迴版本不依賴系統堆疊，而是使用變數或陣列儲存狀態，能避免因遞迴層數過多而崩潰，執行穩定性更高。

   雖然程式邏輯略為複雜，但能支援較大的輸入範圍。

總結：

   非遞迴版本的 Ackermann 函數雖然邏輯較為繁瑣，但能有效解決遞迴版本的堆疊限制與函式呼叫開銷問題。

   這種設計方式展現了如何用迴圈模擬遞迴運算，有助於理解「遞迴與迭代的等價轉換」概念，同時提升程式的穩定性與可控制性。
